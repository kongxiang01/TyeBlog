今天我让 Codex 下载了一个叫 `find-skills` 的 Skill。

一开始我对这件事的想象很接近安装一个 npm 包：下载一组代码，引入一些依赖，然后多出一项可以直接调用的能力。但实际看了一遍目录和安装过程之后，我发现 Skill 更像是一份会被 Agent 按需阅读的工作说明。

它不一定增加新的执行能力。很多时候，它只是让已有能力沿着更清楚的路径工作。

## 下载下来的东西比预想中少

`find-skills` 来自 `vercel-labs/skills` 仓库中的 `skills/find-skills` 子目录。安装完成后，本地目录只有一个文件：

```text
~/.codex/skills/
└── find-skills/
    └── SKILL.md
```

没有 JavaScript，没有 Python 脚本，也没有新的模型文件。

`SKILL.md` 的开头是一段元信息：

```yaml
---
name: find-skills
description: Helps users discover and install agent skills...
---
```

下面则是一份 Markdown 写成的工作流。它告诉 Agent：什么时候应该搜索 Skill，如何查看排行榜，如何使用 `npx skills find`，推荐之前要检查哪些信号，以及用户确认后如何安装。

<Callout title="MY FIRST CORRECTION">
    Skill 不等于可执行插件。它首先是一份 Agent 能够理解的说明书。
</Callout>

## 安装过程其实很朴素

这次安装不是直接运行 `npx skills add`，而是使用 Codex 内置的 `skill-installer`。沿着安装脚本看下去，过程大致是：

```text
收到安装请求
  ↓
读取 skill-installer 的 SKILL.md
  ↓
调用 install-skill-from-github.py
  ↓
下载 GitHub 仓库 main 分支的 ZIP
  ↓
定位 skills/find-skills 子目录
  ↓
检查目录中是否存在 SKILL.md
  ↓
复制到 ~/.codex/skills/find-skills
  ↓
删除临时文件
```

安装器还做了一些边界检查：

- Skill 路径必须是仓库内部的相对路径。
- 解压 ZIP 时不能把文件写到临时目录之外。
- 选中的目录必须包含 `SKILL.md`。
- 本地已经存在同名 Skill 时，不会直接覆盖。
- ZIP 下载失败时，可以回退到 Git sparse checkout。

这些检查不复杂，但它们回答了一个重要问题：安装 Skill 并不是把整个仓库不加区分地放进本地，而是从仓库中挑出一个经过基本校验的子目录。

## Skill 可以只有说明，也可以带工具

`find-skills` 是最小结构。一般来说，一个 Skill 还可以继续携带脚本、参考资料和模板：

```text
my-skill/
├── SKILL.md        # 必需：名称、触发条件和工作流
├── scripts/        # 可选：Python、Node.js 或 Shell 脚本
├── references/     # 可选：规范、示例和详细文档
├── assets/         # 可选：模板、图片和静态资源
├── agents/         # 可选：Agent 配置
└── LICENSE.txt     # 可选：许可证
```

这让我意识到，Skill 的形态其实有一个很宽的范围。最轻的时候，它是一页 Markdown；更复杂的时候，它可以是一套带脚本和资料的任务工具箱。

<CodeNote label="A USEFUL DISTINCTION">
    `SKILL.md` 描述应该怎么做；`scripts/` 中的文件才负责在需要时真正执行动作。
</CodeNote>

## Skill 的触发过程

安装完成并重启 Codex 之后，系统不会在每次对话中完整读取所有 Skill。更合理的过程是先加载摘要，再按任务需要打开具体说明：

```text
Codex 启动
  ↓
扫描用户 Skill 和系统 Skill 目录
  ↓
读取各个 SKILL.md 的 name 与 description
  ↓
把 Skill 摘要放入会话上下文
  ↓
用户提出一个具体需求
  ↓
Agent 判断某个 Skill 与任务匹配
  ↓
读取该 Skill 的完整 SKILL.md
  ↓
按照说明调用 CLI、脚本、MCP 或 API
```

以 `find-skills` 为例，如果我问“有没有适合 React 性能优化的 Skill”，Agent 会先根据描述判断它与任务匹配，然后读取完整说明。接下来，它可能查看 `skills.sh` 排行榜，或者执行：

```bash
npx skills find react performance
```

找到结果之后，它还应该检查安装量、来源可信度和 GitHub 仓库情况，而不是直接推荐第一个搜索结果。

这里有一个我之前没有注意到的分层：Skill 自身并不一定完成任务。它可以继续引导 Agent 使用 CLI、脚本、外部服务或者其他工具。

## 我现在怎样理解 Skill

顺着这次安装看下来，我更愿意把 Skill 理解为 Agent 的“可安装经验”。

模型本身有通用能力，但通用能力不等于稳定流程。一个写得好的 Skill 会补上触发条件、检查顺序、工具入口和风险边界，让 Agent 在面对某类任务时少走弯路。

它和传统插件有交集，但不是同一件事：

| 维度     | 传统插件               | Agent Skill                            |
| -------- | ---------------------- | -------------------------------------- |
| 核心内容 | 可执行代码             | 面向 Agent 的工作流说明                |
| 最小结构 | 通常需要代码和入口     | 一个 `SKILL.md` 即可                   |
| 调用方式 | 程序直接调用函数或命令 | Agent 先理解任务，再决定是否读取和执行 |
| 扩展能力 | 增加新的运行逻辑       | 固化经验，也可以编排已有工具           |

## Skill 和 MCP 也不是一回事

看到 Skill 可以继续调用外部工具之后，我很自然地想到 MCP。两者都会扩展 Agent，但它们解决的问题不同。

MCP（Model Context Protocol）更接近一套连接协议。一个 MCP Server 可以向 Agent 暴露工具、资源和提示模板，让模型以结构化方式访问 GitHub、数据库、浏览器或者内部系统。

Skill 则更像经验层。它告诉 Agent 在什么场景下应该做什么、先检查什么、什么时候调用哪个工具，以及出现异常时如何继续。

| 维度             | Agent Skill                      | MCP                                       |
| ---------------- | -------------------------------- | ----------------------------------------- |
| 主要作用         | 固化任务经验与执行顺序           | 连接外部能力与数据                        |
| 常见载体         | `SKILL.md`，可附带脚本和参考资料 | MCP Server 暴露的 tools、resources 等接口 |
| 触发方式         | Agent 根据任务语义按需读取       | Agent 根据需要调用已注册的结构化接口      |
| 是否独立完成任务 | 不一定，常常继续编排其他工具     | 提供能力入口，但通常不负责完整工作流      |

一个更具体的例子是处理 GitHub PR。Skill 可以要求 Agent 先读取 PR 元信息，再判断是进入代码审查、处理 review comments，还是排查 CI；GitHub MCP 则负责真正返回 PR、issue 和评论数据。

```text
用户提出 GitHub 任务
  ↓
Agent 读取 GitHub Skill，理解处理流程
  ↓
Skill 指引 Agent 调用 GitHub MCP 工具
  ↓
MCP Server 返回结构化仓库数据
  ↓
Agent 按照 Skill 中的步骤继续判断和处理
```

<Callout title="ANOTHER CORRECTION">
    MCP 解决“工具怎样接进来”，Skill 解决“拿到工具之后应该怎样做”。它们更适合组合，而不是互相替代。
</Callout>

当然，Skill 仍然值得谨慎对待。只有 Markdown 的 Skill 风险相对有限，但带有 `scripts/`、安装命令或者外部服务访问的 Skill，需要像检查普通依赖一样检查来源和内容。MCP Server 也需要单独检查权限范围，因为它连接的通常是真实数据和外部系统。

## 留下一个简单检查清单

以后再安装第三方 Skill，我会先看这几件事：

1. `SKILL.md` 要求 Agent 做什么？
2. 是否包含 `scripts/`，脚本会访问哪些文件和网络服务？
3. 是否要求安装额外依赖或执行命令？
4. 仓库作者是否可信，最近是否仍在维护？
5. Skill 是提供建议，还是会修改文件、发布内容或操作外部系统？

我不太想把 Skill 当成下载完就不用管的东西。既然要交给 Agent 使用，至少应该知道它会读取什么、调用哪些工具、执行什么命令，又可能影响哪些文件或外部系统。流程看过一遍，心里才有底。
